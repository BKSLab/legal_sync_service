# 1. Назначение сервиса

## Цель

Legal Sync Service предназначен для автоматического отслеживания изменений нормативных актов и своевременного обновления базы знаний RAG Service.

Сам сервис **не занимается**:

* хранением нормативных актов;
* построением чанков;
* генерацией embeddings;
* индексацией векторной базы;
* обработкой документов для поиска.

Все перечисленные задачи выполняются исключительно RAG Service.

Legal Sync Service отвечает только за:

* ведение списка отслеживаемых нормативных актов;
* мониторинг изменений законодательства через API publication.pravo.gov.ru;
* определение даты вступления изменений в силу;
* получение актуальной редакции нормативного акта;
* извлечение измененной структурной единицы (статьи, пункта);
* передачу данных в RAG Service через REST API.

# 5. Структура данных

## 5.1 Реестр отслеживаемых документов

Хранится список документов, подлежащих мониторингу.

Поля:

* `document_id` — идентификатор документа в RAG
* `short_name` — краткое название для поиска по API (например: «Трудовой кодекс»)
* `full_title` — полное наименование нормативного акта
* `category`
* `audience`
* `topic`
* `source_title`
* `publication_block` — блок публикации на pravo.gov.ru (`president`, `government` и т.д.) *(⚠ см. вопрос #8)*
* `consolidated_source_url` — URL источника актуальной (консолидированной) редакции
* `is_active`

Назначение:

* определяет, какие документы необходимо мониторить;
* содержит значения по умолчанию для передачи в RAG;
* `short_name` используется для фильтрации публикаций по API.

---

## 5.2 Событие изменения

Создается при обнаружении нового закона, изменяющего отслеживаемый документ.

Поля:

* `tracked_document` — ссылка на документ из реестра (5.1)
* `section_number` — номер изменяемой статьи
* `section_title` — заголовок статьи
* `amending_law_ref` — ссылка на закон-поправку (eoNumber или URL)
* `adoption_date` — дата подписания закона-поправки
* `effective_date` — дата вступления изменений в силу
* `change_description` — краткое описание изменений (извлекается из текста)
* `delta_text` — текст поправки из закона-поправки
* `consolidated_text` — извлечённый текст статьи из консолидированной редакции *(заполняется на Этапе 7; хранится временно для review и отправки в RAG — не является постоянным хранилищем документа)*
* `consolidated_text_source` — URL источника, из которого был получен консолидированный текст
* `audience_override` — переопределение audience (если отличается от умолчания)
* `topic_override`
* `source_title_override`

---

## 5.3 Статус обработки события

> ⚠️ **Вопрос #4:** Уточнить, является ли статус частью таблицы 5.2 или отдельной связанной таблицей.

Поля:

* `status` — текущий статус (см. ниже)
* `send_at` — запланированная дата отправки (= `effective_date`)
* `reviewed_by` — кто подтвердил событие
* `reviewed_at` — когда подтверждено
* `review_notes` — примечания оператора
* `sent_at` — фактическая дата отправки
* `rag_response` — ответ RAG Service
* `retry_count` — количество попыток отправки
* `last_error` — текст последней ошибки

### Статусная машина

```
draft ──→ approved ──→ scheduled ──→ processing ──→ sent
                                                 └──→ failed ──→ (retry) ──→ processing
```

| Статус | Кто устанавливает | Когда |
|--------|-------------------|-------|
| `draft` | Система | При автоматическом обнаружении изменения |
| `approved` | Оператор | После ручной проверки в административном интерфейсе |
| `scheduled` | Система | Автоматически при подтверждении; `send_at` = `effective_date` |
| `processing` | Система | В момент начала обработки планировщиком |
| `sent` | Система | После успешной отправки в RAG |
| `failed` | Система | При ошибке отправки |

> ⚠️ **Вопрос #3:** Если ручной review не требуется, статус `draft` можно пропустить и создавать событие сразу в `approved`/`scheduled`. Нужно принять решение.

# 6. Алгоритм работы сервиса

## Этап 1. Добавление документа

Через административный интерфейс оператор добавляет документ в реестр.

Указываются:

* идентификатор документа в RAG;
* `short_name` для поиска по API;
* `publication_block` (блок публикации на pravo.gov.ru);
* источник консолидированной редакции (`consolidated_source_url`);
* категория, тема, аудитория по умолчанию.

После сохранения документ становится доступен для мониторинга.

---

## Этап 2. Мониторинг изменений

Один раз в сутки APScheduler запускает задачу проверки изменений.

### Источник данных

Используется публичный REST API `publication.pravo.gov.ru`.

Аутентификация не требуется. Режим — только чтение.

### Алгоритм

```
1. Запросить все документы, опубликованные сегодня:

   GET /api/Documents
     ?PeriodType=daily
     &Block={publication_block}
     &PageSize=200

2. Для каждого активного документа из реестра:
   - Проверить поля name/complexName каждой публикации
   - Искать вхождение short_name отслеживаемого документа
     (например: «Трудовой кодекс», «Гражданский кодекс»)

3. Если совпадений нет — завершить проверку.

4. Если совпадение найдено:
   - Проверить, не обработан ли этот eoNumber ранее
     (поле amending_law_ref в таблице событий)
   - Если уже обработан — пропустить (идемпотентность)
   - Если новый — перейти к Этапу 3
```

### Поля ответа API, которые используются

| Поле | Назначение |
|------|-----------|
| `eoNumber` | Уникальный ID публикации; используется для скачивания PDF и дедупликации |
| `name` | Название закона; проверяется на вхождение `short_name` |
| `complexName` | Полное составное название; резервная проверка |
| `documentDate` | Дата подписания закона |
| `id` | GUID документа |

---

## Этап 3. Анализ закона-поправки

После обнаружения закона-поправки необходимо получить его текст и извлечь из него:

* список изменяемых статей (номер + заголовок);
* дату вступления изменений в силу;
* краткое описание изменений.

### Получение текста закона-поправки

```
1. Скачать PDF по eoNumber:
   GET publication.pravo.gov.ru/File/GetFile/{eoNumber}
   ⚠️ Вопрос #1: URL паттерн требует подтверждения

2. Извлечь текст из PDF через pdfplumber
```

### Алгоритм извлечения данных из текста

**Извлечение изменяемых статей — приоритет:**

1. Регулярные выражения по конструкциям вида:
   ```
   дополнить статьёй 128.1
   статью 136 изложить в следующей редакции
   в статье 140 слова «...» заменить словами «...»
   ```
2. Структурные правила по заголовкам закона-поправки.
3. LLM — только если regex и правила не дали результата.

**Извлечение даты вступления в силу — приоритет:**

1. Регулярные выражения по типовым формулировкам:
   ```
   Настоящий Федеральный закон вступает в силу с 1 января 2026 года
   вступает в силу по истечении девяноста дней после дня официального опубликования
   вступает в силу со дня официального опубликования
   ```
2. Расчёт даты при формулировках «по истечении N дней» — от `documentDate` из API.
3. LLM — только если предыдущие способы не дали результата.

> Результат работы LLM при определении даты не сохраняется в `sent` автоматически.
> Требуется ручное подтверждение оператором (статус `draft`).

---

## Этап 4. Создание события

По каждой найденной изменяемой статье создаётся отдельная запись события.

Заполняются:

* `tracked_document` — ссылка на документ из реестра;
* `section_number` — номер статьи;
* `section_title` — заголовок статьи (из текста закона-поправки);
* `amending_law_ref` — `eoNumber`;
* `adoption_date` — `documentDate` из API;
* `effective_date` — дата, определённая на Этапе 3;
* `change_description` — краткое описание из текста;
* `delta_text` — фрагмент закона-поправки, касающийся этой статьи;
* `status` = `draft`;
* `send_at` = `effective_date`.

После создания событие отображается в административном интерфейсе для проверки оператором.

---

## Этап 5. Ручная проверка (административный интерфейс)

Оператор проверяет событие в разделе «События изменений»:

* просматривает данные;
* при необходимости корректирует `effective_date`, `section_number`, overrides;
* нажимает «Подтвердить» → статус меняется на `approved`, затем автоматически на `scheduled`.

При нажатии «Отмена» событие исключается из очереди.

---

## Этап 6. Планировщик

APScheduler ежедневно запускает задачу отбора событий для обработки:

```sql
SELECT *
FROM legal_changes
WHERE status = 'scheduled'
  AND send_at <= NOW()
```

Если записи отсутствуют — работа завершается.

Если записи найдены — статус каждой записи меняется на `processing`, начинается Этап 7.

---

## Этап 7. Получение актуальной редакции

В день вступления изменений сервис получает **актуальную (консолидированную) редакцию** нормативного акта по `consolidated_source_url` из реестра.

Важно: сервис никогда не применяет изменения самостоятельно. Он всегда работает только с уже опубликованной официальной редакцией.

Подробный алгоритм получения документа — см. Раздел 7.

---

## Этап 8. Извлечение статьи

Из полного текста консолидированной редакции извлекается полный текст нужной статьи.

Подробный алгоритм — см. Приложение А.

# 7. Стратегия получения документов

Сервис работает с двумя типами документов, каждый из которых имеет свою стратегию получения.

---

## 7.1 Закон-поправка (с publication.pravo.gov.ru)

Источник: API publication.pravo.gov.ru.

```
1. Получить eoNumber из результатов мониторинга (Этап 2)
2. Скачать PDF:
   GET /File/GetFile/{eoNumber}            ⚠️ URL требует подтверждения
3. Извлечь текст через pdfplumber
```

PDF является единственным форматом для документов с publication.pravo.gov.ru.

---

## 7.2 Консолидированная редакция (из consolidated_source_url)

Источник: задаётся при добавлении документа в реестр. Может быть любым правовым порталом.

Стратегия получения — каскадная:

```
Шаг 1. Попытка прямого HTTP-запроса (httpx):
        GET consolidated_source_url
        → если в ответе HTML с текстом документа → парсинг через BeautifulSoup4/lxml
        → если в ответе PDF → извлечение через pdfplumber
        → если в ответе DOCX → извлечение через python-docx

Шаг 2. Если Шаг 1 не дал текст (JS-рендеринг, редирект, форма):
        → Playwright: открыть страницу в браузере, дождаться загрузки,
          извлечь текст документа из DOM

Шаг 3. Если Шаг 2 вернул ссылку на файл (PDF/DOCX):
        → скачать файл, извлечь текст
```

### Приоритет форматов

| Приоритет | Формат | Инструмент |
|-----------|--------|-----------|
| 1 | HTML с семантической структурой | BeautifulSoup4 / lxml |
| 2 | DOCX | python-docx |
| 3 | PDF | pdfplumber |

Использование PDF допускается только при отсутствии структурированного текста.

### Когда необходим Playwright

* Сайт требует JavaScript для рендеринга содержимого.
* Доступ к документу требует навигации по формам (выбор даты редакции).
* Прямой HTTP-запрос возвращает пустой или неполный HTML.

# 8. Отправка в RAG

После успешного извлечения статьи формируется запрос:

```
PUT /document/{document_id}/sections/{section_number}
```

Передаются:

* `category`
* `raw_text`
* `section_title`
* `version`
* `effective_date`
* `audience`
* `topic`
* `source_title`

Значения берутся из реестра документов; при наличии override-полей в событии — используются они.

После успешной отправки:

* сохранить ответ RAG в `rag_response`;
* изменить статус на `sent`;
* записать `sent_at`.

При ошибке:

* увеличить `retry_count`;
* сохранить `last_error`;
* изменить статус на `failed`.

> ⚠️ **Вопрос #5:** Нужно определить стратегию повторных попыток: максимальное число попыток, интервал (рекомендуется экспоненциальный backoff с ограничением).

# 10. Логирование

Все действия должны журналироваться через loguru.

Минимальный набор событий:

* начало задачи мониторинга;
* запрос к API publication.pravo.gov.ru (URL, статус ответа);
* найдено изменение (документ, eoNumber);
* создано событие (document_id, section_number, effective_date);
* начало получения консолидированной редакции (URL, метод);
* документ получен (формат, размер);
* статья извлечена (section_number, длина текста);
* отправка в RAG (document_id, section_number);
* успешное обновление (rag_response);
* ошибка (тип, сообщение, контекст);
* повторная попытка (retry_count).

# 11. Требования к надёжности

* PostgreSQL является единственным источником истины.
* Перезапуск приложения не должен приводить к потере событий или дублированию обработки.
* Все операции должны быть идемпотентными: повторный запуск мониторинга не создаёт дубликаты событий (проверка по `eoNumber`).
* Повторная отправка одной статьи не должна приводить к повреждению данных в RAG.
* Любая ошибка должна сохраняться в журнале и отображаться в административном интерфейсе.
* При обработке нескольких статей одного закона-поправки ошибка в одной статье не должна блокировать обработку остальных.

# 12. Будущее развитие

Архитектура должна предусматривать возможность:

* подключения новых источников нормативных актов (новые `publication_block`);
* добавления новых категорий документов;
* ручного запуска мониторинга и обработки из административного интерфейса;
* повторной отправки статьи в RAG вручную;
* поддержки нескольких источников консолидированных текстов для одного документа;
* подключения дополнительных модулей анализа изменений без изменения существующей архитектуры;
* поддержки не-статейных структурных элементов (пунктов, приложений) *(⚠ вопрос #6)*.

# Приложение А. Алгоритм извлечения статьи из нормативного акта

## Цель

После наступления даты вступления изменений в силу Legal Sync Service должен получить **полную актуальную редакцию конкретной статьи** нормативного акта из консолидированного источника и передать её в RAG Service.

Сервис **не изменяет** текст нормативного акта и **не применяет** поправки самостоятельно. Источником данных всегда является уже опубликованная официальная консолидированная редакция документа.

---

## Входные данные

На момент запуска задачи известны:

* `document_id` — идентификатор документа в RAG;
* `section_number` — номер статьи;
* `section_title` — ожидаемый заголовок статьи;
* `effective_date` — дата вступления изменений в силу;
* `consolidated_source_url` — источник получения актуальной редакции.

---

## Общий алгоритм

```
Получить актуальную редакцию документа (см. Раздел 7.2)
↓
Преобразовать документ в текст
↓
Нормализовать текст
↓
Найти начало нужной статьи
↓
Проверить совпадение заголовка
↓
Найти начало следующей статьи
↓
Извлечь текст между ними
↓
Проверить корректность извлечения
↓
Очистить служебные элементы
↓
Передать статью в RAG Service
```

---

## Шаг 1. Получение документа

Выполняется по алгоритму из Раздела 7.2.

---

## Шаг 2. Преобразование в текст

Документ приводится к обычному тексту.

Необходимо удалить:

* номера страниц;
* колонтитулы;
* повторяющиеся заголовки;
* служебные ссылки;
* навигационные элементы сайта;
* лишние переносы строк;
* множественные пробелы.

Результатом должен быть единый текст нормативного акта.

---

## Шаг 3. Нормализация

Перед поиском статьи необходимо привести текст к единому виду:

* замена неразрывных пробелов (` `) на обычные;
* замена различных видов тире на стандартный символ;
* удаление двойных пробелов;
* унификация переводов строк (`\r\n` → `\n`);
* удаление скрытых Unicode-символов (`​`, `﻿` и др.).

---

## Шаг 4. Поиск начала статьи

Приоритет способов поиска:

1. **HTML DOM** — если источник предоставляет структурированный HTML:
   - якоря вида `#article-128`, `#st128`;
   - атрибуты `id` на элементах;
   - CSS-классы, специфичные для портала.

2. **Регулярные выражения** — основной способ для текста:

```python
import re

pattern = re.compile(
    rf"^Статья\s+{section_number}\b.*?$",
    re.MULTILINE | re.IGNORECASE
)
match = pattern.search(document_text)
```

Учитываемые варианты написания:

```
Статья 128
Статья 128.
Статья 128. Отпуск без сохранения заработной платы
СТАТЬЯ 128
```

После нахождения совпадения проверить:

* совпадает ли номер статьи;
* совпадает ли заголовок (если `section_title` задан).

---

## Шаг 5. Определение границ статьи

Начало статьи — найденный заголовок.

Конец статьи — начало следующей статьи:

```python
next_pattern = re.compile(
    r"^Статья\s+\d+\b",
    re.MULTILINE
)
```

Поиск следующей статьи начинается с позиции после заголовка текущей.

Для последней статьи документа концом считается конец файла.

---

## Шаг 6. Проверка корректности

После извлечения выполнить автоматические проверки:

* статья не пустая;
* длина текста превышает минимальный порог *(⚠ порог требует определения)*;
* номер статьи соответствует ожидаемому;
* заголовок совпадает (если задан);
* текст не содержит элементов следующей статьи.

Если проверка не пройдена — статья считается не извлечённой, событие переходит в статус `failed` с указанием причины.

---

## Шаг 7. Очистка статьи

Перед отправкой в RAG удалить:

* лишние пустые строки;
* пробелы в начале и конце строк;
* служебные ссылки;
* номера страниц;
* HTML-разметку;
* элементы навигации.

Сохранить:

* структуру статьи;
* абзацы;
* нумерацию пунктов;
* подпункты;
* перечни.

Текст нормативного акта изменять запрещено.

---

## Использование LLM

LLM **не используется** для извлечения статьи из консолидированной редакции.

Допускается применение только как резервного механизма при полной невозможности определить структуру документа программными средствами.

LLM запрещается:

* определять границы статьи самостоятельно;
* изменять или восстанавливать текст статьи;
* применять поправки.

Результат работы LLM не отправляется в RAG автоматически. Требуется ручное подтверждение оператором.

---

## Приоритет способов извлечения

| Приоритет | Метод |
|-----------|-------|
| 1 | HTML DOM (якоря, id, CSS-классы) |
| 2 | XPath / CSS-селекторы |
| 3 | Регулярные выражения |
| 4 | Правила на основе структуры документа |
| 5 | LLM (только как последний резервный вариант) |

---

## Требования к реализации

* Алгоритм должен быть полностью детерминированным.
* Один и тот же документ должен всегда давать одинаковый результат.
* Извлечённая статья не должна модифицироваться.
* Любая ошибка должна журналироваться.
* При невозможности извлечения событие должно переходить в статус `failed` с сохранением причины ошибки.

# Приложение Б. Алгоритм анализа закона-поправки

## Цель

Из текста закона-поправки (PDF с publication.pravo.gov.ru) извлечь:

1. Список изменяемых статей (номер + заголовок).
2. Дату вступления изменений в силу.
3. Краткое описание изменений по каждой статье.

---

## Входные данные

* `eoNumber` — номер электронного опубликования закона-поправки.
* `adoption_date` — дата подписания (из API поля `documentDate`).

---

## Алгоритм

```
1. Скачать PDF: /File/GetFile/{eoNumber}  ⚠️ URL требует подтверждения
2. Извлечь текст через pdfplumber
3. Нормализовать текст (см. Приложение А, Шаг 3)
4. Извлечь изменяемые статьи
5. Извлечь дату вступления в силу
6. Сформировать описание изменений
```

---

## Извлечение изменяемых статей

**Приоритет 1 — Регулярные выражения** по типовым формулировкам:

```python
patterns = [
    r"статью\s+(\d+[\.\d]*)\s+(?:изложить|дополнить|признать|исключить)",
    r"в\s+статье\s+(\d+[\.\d]*)",
    r"дополнить\s+статьёй\s+(\d+[\.\d]*)",
    r"статья\s+(\d+[\.\d]*)\s+утрачивает силу",
]
```

**Приоритет 2 — Структурные правила:**

Законы-поправки, как правило, структурированы по статьям самого закона-поправки:

```
Статья 1
Внести в Трудовой кодекс Российской Федерации следующие изменения:
1) статью 128 изложить в следующей редакции: ...
2) в статье 136 слова «...» заменить словами «...»

Статья 2
Настоящий Федеральный закон вступает в силу ...
```

Поиск блоков `Статья N` в тексте закона-поправки с последующим анализом содержимого.

**Приоритет 3 — LLM:**

Применяется только если regex и правила не дали результата. Результат требует ручного подтверждения оператором.

---

## Извлечение даты вступления в силу

**Приоритет 1 — Регулярные выражения** по типовым формулировкам:

```python
# Конкретная дата
r"вступает в силу\s+(?:с\s+)?(\d{1,2}\s+\w+\s+\d{4})"

# По истечении N дней
r"по истечении\s+(\w+)\s+дней?\s+(?:после|со дня)\s+(?:дня\s+)?официального опубликования"

# Со дня опубликования
r"со дня\s+(?:его\s+)?официального опубликования"
```

**Приоритет 2 — Расчёт:**

При формулировках «по истечении N дней» дата вычисляется от `adoption_date + N`.

**Приоритет 3 — LLM:**

Только если предыдущие способы не дали результата. Требует подтверждения.

**Fallback:**

Если дату вступления определить не удалось программно — событие создаётся со статусом `draft` и пустым `effective_date`. Оператор заполняет вручную.

---

## Результат

Для каждой изменяемой статьи формируется:

```json
{
  "section_number": "128",
  "section_title": "Отпуск без сохранения заработной платы",
  "effective_date": "2026-01-01",
  "delta_text": "статью 128 изложить в следующей редакции: ...",
  "change_description": "Статья изложена в новой редакции"
}
```
